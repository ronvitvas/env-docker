# Банк идей UDS v2: руководство по модулю

Дата актуализации: 2026-05-01.  
Модуль: `uds.ideabank2`.  
Раздел портала: `/ideabank/`.

Документ описывает актуальное состояние модуля после последних доработок: публичный раздел, роли, настройки, путь идеи, модерацию, коины, конкурсы, демо-данные, установку и удаление. Текст ориентирован на бизнес-пользователя и администратора портала; технические детали приведены только там, где они помогают сопровождать раздел.

## 1. Назначение модуля

`uds.ideabank2` реализует корпоративный раздел “Банк идей”: сотрудники подают инициативы, получают поддержку коллег, проходят модерацию и дальнейший маршрут рассмотрения, а за значимые события получают коины. Руководители и кураторы используют раздел для управления очередью идей, конкурсами, челленджами, новостями, статистикой и рейтингами.

Основные возможности:

- подача идеи через форму с черновиком;
- публичный реестр идей с режимами “все”, “мои”, “черновики”, “лучшие практики”, “модерация”;
- карточка идеи с описанием, авторами, маршрутом, обсуждением, обратной связью, экспертными оценками и решением комитета;
- модерация и смена статусов;
- начисление коинов, баланс, история и лидерборд;
- конкурсы идей и тематические челленджи;
- новости, документы, статистика и аллея славы;
- расширенные демо-данные для полноценного просмотра раздела за несколько лет.

## 2. Состав публичного раздела

Публичный контур расположен в `/ideabank/`. При установке модуля установщик создаёт страницы, меню, компоненты, CSS/JS assets и общий helper публичного интерфейса.

| Страница | Назначение |
|---|---|

| `/ideabank/management.php` | Список идей: фильтры, режимы просмотра, карточки идей, переходы к деталям. |
| `/ideabank/ppu-form.php` | Форма подачи или редактирования идеи. |
| `/ideabank/ppu-detail.php?id=...` | Детальная карточка идеи. |
| `/ideabank/news.php` | Список новостей банка идей. |
| `/ideabank/news-detail.php?id=...` | Детальная новость. |
| `/ideabank/contests.php` | Список конкурсов идей. |
| `/ideabank/contest-detail.php?id=...` | Детальная карточка конкурса. |
| `/ideabank/docs.php` | Документы, шаблоны, материалы и подсказки. |

В актуальной версии публичная оболочка приведена к плановому виду: стандартный заголовок Bitrix используется как основной, хлебные крошки скрыты, внутренний дублирующий заголовок и лишние CTA в верхней зоне убраны.

## 3. Роли и права

Модуль использует пять служебных групп. Установщик создаёт их без дублей, сохраняет ID в настройках модуля и назначает права.

| Роль | Группа | Право модуля | Назначение |
|---|---|---:|---|
| Участники | `UDS_IDEABANK2_PARTICIPANTS` | `R` | Просмотр публичного раздела, участие в банке идей. |
| Модераторы | `UDS_IDEABANK2_MODERATORS` | `W` | Проверка идей, смена статусов, работа с очередью модерации. |
| Эксперты | `UDS_IDEABANK2_EXPERTS` | `W` | Экспертная оценка идей при включённой экспертизе. |
| Комитет | `UDS_IDEABANK2_COMMITTEE` | `W` | Принятие решений комитетом при включённом контуре комитета. |
| Администраторы | `UDS_IDEABANK2_ADMINS` | `W` | Полное администрирование раздела. |

Публичный интерфейс дополнительно учитывает feature flags. Например, пользователь может состоять в группе модераторов, но очередь модерации не будет доступна, если выключен флаг `feature_moderation`.

## 5. Путь идеи

Путь идеи начинается с создания карточки и заканчивается внедрением, отклонением, помещением в бэклог или другой финальной развилкой. Маршрут фиксируется в истории workflow, а обратная связь сохраняется отдельно.

### 5.1. Создание и черновик

Участник открывает `/ideabank/ppu-form.php` и заполняет форму. При создании идея получает:

- автора-владельца;
- запись автора с долей 100%;
- статус черновика на уровне workflow;
- дату создания;
- признак `IS_DRAFT = Y`, если пользователь сохраняет черновик.

Черновик виден в режиме “Черновики”, если включён `feature_drafts`. Автор может редактировать свой черновик. После отправки редактирование ограничивается настройкой `feature_edit_after_submit`.

### 5.2. Отправка идеи

Когда автор отправляет идею:

- `IS_DRAFT` меняется на `N`;
- статус переводится в `moderation`;
- заполняется `SUBMITTED_AT`;
- в workflow добавляется запись “Идея отправлена на модерацию”;
- запускается начисление коинов за событие `submitted`.

Проверить в интерфейсе: `/ideabank/management.php?mode=mine` и `/ideabank/management.php?mode=moderation` для модератора.

### 5.3. Основные статусы

| Код | Бизнес-смысл |
|---|---|
| `moderation` | Идея ожидает первичной проверки. |
| `published` | Идея опубликована для обсуждения. |
| `initial_review` | Идёт первичная проверка эффекта и применимости. |
| `kpu` | Идея передана на КПУ или аналогичный этап оценки. |
| `transferred` | Идея передана в проектный контур. |
| `accepted` | Идея принята в работу. |
| `implementation` | Идёт реализация или пилот. |
| `implemented` | Идея реализована. |
| `deployed` | Практика внедрена и готова к тиражированию. |
| `revise` | Идея требует доработки. |
| `backlog` | Идея отложена до появления ресурсов или условий. |
| `rejected` | Идея отклонена. |
| `duplicate` | Идея признана дубликатом. |

Смена статуса через сервис добавляет запись workflow, обратную связь и при статусах `accepted`/`implemented` запускает начисления коинов.

## 6. Модерация идей

Модерация предназначена для первичной проверки идеи: полнота описания, корректность, отсутствие явных дублей, понятность проблемы и предложения.

### 6.1. Участники процесса

- Автор подаёт идею.
- Модератор видит очередь, если состоит в группе модераторов или является администратором и включён `feature_moderation`.
- Администратор может выполнять действия с расширенными правами.

### 6.2. Решения модератора

Модераторское решение выполняется через действие `moderate` и принимает один из вариантов:

| Решение | Что происходит |
|---|---|
| `publish` | Идея переводится в `published`, заполняются данные модерации и дата публикации. |
| `revise` | Идея переводится в `revise`, автор получает обратную связь для доработки. |
| `reject` | Идея переводится в `rejected`, автор получает причину отклонения. |

При любом решении сохраняются:

- ID модератора;
- дата модерации;
- решение;
- текст обратной связи;
- запись в workflow;
- запись feedback с тоном `success`, `warning` или `danger`.

Проверить в интерфейсе: `/ideabank/management.php?mode=moderation`, карточка идеи `/ideabank/ppu-detail.php?id=...`, админская страница модерации.

### 6.3. Смена статуса вне первичной модерации

Модератор или администратор может менять статус идеи на последующих этапах. При смене статуса:

- обновляется `STATUS_ID` и `STAGE`;
- добавляется запись workflow;
- добавляется feedback для автора;
- при `accepted` начисляются коины за принятие;
- при `implemented` начисляются коины за реализацию.

## 7. Коины и награды

Коины — внутренняя единица поощрения авторов идей. Они используются для вовлечения, рейтингов и будущих reward-сценариев.

### 7.1. События начисления

Базовые события:

| Событие | Когда возникает | Базовое правило установки |
|---|---|---:|
| `submitted` | Идея отправлена на модерацию. | 30 коинов |
| `accepted` | Идея принята в работу. | 45 коинов |
| `implemented` | Идея реализована. | 120 коинов |
| `engagement` | Идея получила вовлечение коллег. | 12 коинов |
| `replicated` | Идею взяли за основу или тиражировали. | 80 коинов |

Фактическое начисление берёт сумму из правил наград. Если правило отсутствует, сервис возвращает ошибку “Правило награды не найдено”. Если сумма правила меньше или равна нулю, начисление пропускается без ошибки.

### 7.2. Где хранятся и отображаются коины

Каждая операция сохраняется в ledger коинов с пользователем, идеей, событием, суммой, описанием и датой. Баланс считается как сумма всех операций пользователя. История показывает последние операции, а лидерборд агрегирует сумму коинов по пользователям.

Пользователь видит:

- баланс на главной;
- историю начислений;
- правила наград;
- место в рейтинге, если включён `feature_leaderboard`.

Администратор управляет правилами на странице “Награды”.

### 7.3. Ограничения текущей версии

- Нет завершённого пользовательского магазина покупок за коины.
- Ручные начисления зарезервированы флагом, но отдельный полноценный интерфейс ручных операций не описан как завершённый бизнес-процесс.
- Списания технически поддерживаются ledger-подходом через отрицательные значения, но публичный сценарий магазина/заказа не завершён.

## 8. Конкурсы и челленджи

Конкурсы и челленджи помогают собирать идеи под конкретную бизнес-задачу: безопасность, экономия, цифровизация, качество, тиражирование практик.

### 8.1. Конкурс идей

Конкурс содержит:

- название;
- описание;
- текстовый срок или период;
- deadline;
- организатора;
- изображение;
- требования к идеям.

Администратор создаёт и редактирует конкурсы в админке. Публично конкурсы отображаются на `/ideabank/contests.php`, а карточка конкурса доступна по `/ideabank/contest-detail.php?id=...`.

Идея может быть связана с конкурсом через участие. В сервисе есть добавление участника конкурса с пользователем и ID идеи.

### 8.2. Челленджи

Челленджи используются как тематические вызовы на главной странице:

- название;
- период;
- цель;
- бонус;
- бизнес-направление;
- подсказки.

Они легче конкурса: помогают направить поток идей, но не заменяют полноценную конкурсную карточку.

## 9. Новости, документы, аллея славы и статистика

### 9.1. Новости

Новости показывают результаты и события банка идей: успешные практики, запуски конкурсов, изменения процесса. У новости есть категория, дата, заголовок, краткое описание, тело, изображения и цитата.

### 9.2. Документы

Страница документов даёт пользователям материалы и подсказки: как оформить сильную идею, какие данные подготовить, куда обращаться за поддержкой. Текущая публичная витрина формируется компонентом документов.

### 9.3. Аллея славы

Аллея славы показывает рейтинги авторов и практик. Основной источник рейтинга — данные коинов и идей. Отображение зависит от `feature_leaderboard` и доступных данных.

### 9.4. Статистика

Статистика показывает:

- количество идей;
- идеи по статусам;
- идеи по категориям;
- воронку этапов;
- долю реализации;
- эффект;
- средний срок модерации;
- срезы по всем идеям или своим идеям.

Страница полезна руководителям, модераторам и владельцам процесса для оценки зрелости банка идей.

## 11. Установка, обновление и удаление

### 11.1. Что создаёт установщик

При установке модуль выполняет полный набор действий:

- регистрирует модуль в Bitrix;
- создаёт таблицы `b_uds_ideabank_*`;
- устанавливает настройки по умолчанию;
- создаёт роли-группы и назначает права;
- добавляет базовые статусы, категории и правила наград;
- при явном флаге устанавливает demo data;
- регистрирует события модуля и admin menu handler;
- копирует admin-прокси файлы в `/bitrix/admin`;
- устанавливает публичные компоненты в `/local/components/uds/ideabank.*`;
- устанавливает `/local/js/uds/ideabank2/public.css|public.js`;
- устанавливает публичный helper `/local/php_interface/include/ideabank2_public_helpers.php`;
- создаёт страницы и меню `/ideabank/`.

Существующие публичные страницы не перезаписываются: установщик создаёт недостающие файлы, чтобы не стереть ручные правки.

### 11.2. Обновление

Обновление повторно гарантирует наличие:

- настроек;
- групп и option-связок;
- публичных страниц;
- компонентов;
- публичных CSS/JS assets;
- admin-файлов;
- меню.

Это важно для случаев, когда модуль уже установлен, а новый пакет добавляет публичный контур или установочные файлы.

### 11.3. Удаление

При удалении модуль показывает выбор:

| Режим | Что удаляется | Что сохраняется |
|---|---|---|
| Без удаления БД | События, admin-файлы, компоненты, публичные CSS/JS assets, сгенерированные страницы, права модуля. | Таблицы БД, настройки, пользовательские данные. |
| Полная очистка | Таблицы БД, настройки, пустые служебные группы, а также файлы из обычного удаления. | Служебные группы с пользователями не удаляются автоматически. |

Файлы публичного раздела удаляются только если они были сгенерированы установщиком и содержат служебный маркер. Это защищает ручные страницы от удаления.

## 12. Demo/test данные

Демо-данные устанавливаются только явно:

- через флаг установки `install_demo_data=Y`;
- или запуском `local/modules/uds.ideabank2/tools/seed_demo.php`.

Актуальный seeder расширен до многолетнего набора и покрывает полноценный обзор модуля:

- идеи за 2022-2026 годы;
- разные категории и бизнес-направления;
- почти все статусы маршрута;
- черновики, анонимные идеи, дубликаты, отклонённые и внедрённые идеи;
- авторов и соавторов;
- комментарии и реакции;
- начисления коинов;
- workflow и feedback;
- экспертные оценки и решения комитета;
- новости, конкурсы, участия в конкурсах и челленджи.

Seeder идемпотентен по кодам идей и названиям справочных сущностей: повторный запуск досоздаёт недостающее, а не обязан полностью пересоздавать базу.

## 13. Debug-auth для разработки

Маршрут `/ideabank/debug_auth.php` предназначен только для локального dev/test E2E.

Защитные условия:

- инструмент выключен по умолчанию;
- требуется `debug_auth_enabled = Y`;
- пользователь должен быть указан в `debug_auth_allowed_user_ids`;
- используется whitelist ID пользователей;
- пароль не принимается;
- redirect ограничен безопасным путём внутри `/ideabank/`;
- страница подключается как standalone route без стандартного Bitrix header, чтобы не попадать в login redirect.

На production debug-auth должен оставаться выключенным.

## 15. Ограничения и зоны развития

- Магазин за коины пока не является завершённым пользовательским процессом.
- `feature_voting`, ручные начисления и часть shop-флагов зарезервированы под развитие.
- Legacy-флаги `enable_expert_review` и `enable_committee` существуют рядом с feature flags `feature_expertise` и `feature_committee`; в публичном контуре ключевыми являются feature flags.
- Таблицы создаются MySQL-ориентированным DDL, поэтому при переносе на другую СУБД потребуется отдельная адаптация схемы.
- Модуль `uds.ideabank2` является основным текущим банком идей. Старый `uds.ideabank` не описывается как основной контур.
