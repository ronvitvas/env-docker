# Mandatory E2E Before Completion

Перед тем как сообщать пользователю, что задача «готова», обязательно выполнить проверку результата.

## Обязательные шаги

1. После каждой правки PHP-файла выполнить `php -l` для изменённых файлов.
2. Для пользовательских страниц/компонентов выполнить runtime-проверку:
   - открыть целевой URL в браузере,
   - убедиться, что нет фаталов/500,
   - проверить базовую работоспособность UI.
3. Если исправлена одна ошибка, продолжать проверку до следующей стабильной точки:
   - нет новых фаталов,
   - страница отвечает корректно,
   - ключевой пользовательский сценарий выполняется.
4. Только после этого давать статус «готово».

## Авторизация в E2E (обязательно)

1. Если для проверки требуется вход в систему и пользователь уже предоставил тестовые доступы, ассистент обязан выполнить авторизацию и продолжить проверку сценариев.
2. Отсутствие авторизации **не является уважительной причиной** для отказа от тестирования, если логин/пароль уже переданы пользователем.
3. Запрещено завершать задачу формулировкой «не удалось протестировать из-за доступа», пока не выполнена хотя бы одна попытка входа с предоставленными тестовыми данными.
4. Если вход не удался, ассистент обязан зафиксировать точную причину (например: неверные данные, 2FA, CAPTCHA, блокировка, техническая ошибка), приложить артефакты (скриншот, console/network) и только после этого запрашивать уточнения.
5. Тестовые учётные данные можно использовать только для runtime-проверки и нельзя записывать в репозиторий, код, правила проекта или логи в открытом виде.

## Запрещено

- Завершать задачу только по одному частичному исправлению без повторного end-to-end теста.
- Игнорировать доступные инструменты проверки (браузер, логи, runtime-проверки).
- Ссылаться на «нет доступа» при наличии переданных тестовых учётных данных без попытки авторизации.
